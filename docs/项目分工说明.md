# 直播打赏微服务系统 — 四人分工说明

> 课程：JavaEE架构与应用课程设计  
> 技术栈：Spring Boot 3、Spring Cloud、MySQL、Nacos  
> 范围：**仅后端**，不含前端

---

## 一、项目背景与目标

模拟 100 名主播、约 30 万观众的直播打赏场景。系统需完成打赏入账结算、明细记录、观众画像、主播 TOP10 打赏观众查询、分成比例管理、可领取余额查询，以及运营多维分析（小时 × 主播 × 性别）。

### 非功能要求（全员需遵守）

| 要求 | 说明 |
|------|------|
| 数据不丢不重 | 打赏记录需幂等控制，财务同步需可靠 |
| 高可用 | 依赖服务不可达时，观众侧仍可降级服务 |
| 链路追踪 | 各服务记录 HTTP 请求参数，日志通过 `traceId` 关联 |
| 快速启动 | 避免长时间预热 |
| 多节点 | 各服务支持水平扩缩容 |

### 3.2 数据库设计要点

1. **打赏幂等**：`tip_record.tip_no` 唯一索引，防止重复入账
2. **分成比例历史**：`anchor_commission_rate` 按生效时间记录，结算时按打赏时刻匹配比例
3. **财务同步状态**：`tip_record.finance_sync_status` 标记是否已同步财务
4. **画像与统计**：`audience_tip_summary`、`tip_hourly_stat` 供经营分析使用
5. **链路日志**：`http_request_log` 存储各服务 HTTP 请求快照

### 3.3 公共模块规范

- 包名：`com.live.tip.common.*`
- 统一返回：`Result<T>`（code / message / data / traceId）
- 业务异常：`BusinessException`
- 常量：服务名、Header 名（`X-Trace-Id`）
- 枚举：`GenderEnum`、`ProfileLevelEnum`、`FinanceSyncStatusEnum`

## 四、成员二：观众服务（audience-service）

### 4.1 核心职责

观众侧所有接口的实现与打赏主流程编排。

### 4.2 接口清单

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/audience/tips` | 观众打赏（幂等，写 `tip_record`） |
| GET | `/api/audience/profile/{audienceId}` | 查询观众标签画像 |
| GET | `/api/audience/anchors/{anchorId}/top-tippers` | 某主播打赏 TOP10 观众 |
| POST | `/api/audience/tips/sync-finance` | 将待同步打赏推送给财务服务（内部/定时） |

### 4.3 详细要求

1. **打赏接口**
   - 请求体含：`tipNo`（幂等键）、`anchorId`、`audienceId`、`amount`
   - 使用数据库唯一约束 + 业务校验，保证不丢不重
   - 写入成功后异步/同步调用 finance-service 结算
   - 记录 HTTP 请求日志（含 traceId）

2. **观众画像查询**
   - 通过 OpenFeign 调用 analytics-service
   - **2 秒超时**：超时或异常时返回友好提示（不抛 500）
   - 集成 **Resilience4j 熔断**：某 analytics 节点异常时熔断该实例

3. **TOP10 查询**
   - 按 `anchor_id` 聚合 `tip_record`，取累计打赏金额前 10 名观众姓名
   - 支持分页参数，默认返回 10 条

4. **财务同步**
   - 扫描 `finance_sync_status = 0` 的记录，批量推送 finance-service
   - 推送成功更新状态为已同步；失败保留待重试

### 4.4 技术要点

- MyBatis-Plus 访问 `tip_record`、`audience`、`anchor`
- OpenFeign + LoadBalancer 调用 finance / analytics
- Resilience4j（熔断、超时）
- 全局异常处理继承 common 规范

### 4.5 验收标准

- [ ] 同一 `tipNo` 重复提交不产生重复记录
- [ ] 画像查询 2 秒内返回或友好降级
- [ ] TOP10 数据与 SQL 聚合结果一致
- [ ] 日志中可凭 traceId 串联一次打赏全链路

## 五、成员三：财务服务（finance-service）

### 5.1 核心职责

打赏接收、分成结算、比例配置、可领取余额查询。

### 5.2 接口清单

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/finance/settlements/receive` | 接收 audience 推送的打赏并结算 |
| PUT | `/api/finance/commission-rates` | 设置某主播分成比例（写入历史表） |
| GET | `/api/finance/anchors/{anchorId}/balance` | 查询主播可提取打赏余额 |

### 5.3 详细要求

1. **打赏结算**
   - 根据打赏时刻查询 `anchor_commission_rate` 生效记录，计算平台/主播分成
   - 写入 `finance_settlement`，更新 `anchor_balance`
   - 接口需幂等：同一 `tipId` 不重复结算

2. **分成比例设置**
   - 请求：`anchorId`、`commissionRate`（平台抽成比例 0~1）、`effectiveTime`
   - 新比例仅对生效时间之后的打赏生效；历史打赏按旧比例

3. **可领取余额**
   - `available_amount = total_anchor_amount - withdrawn_amount`
   - 返回：累计打赏、累计分成、已提取、可提取

### 5.4 技术要点

- 事务保证 settlement + balance 一致性
- 分成计算抽取为独立 Service 方法，便于单测
- 记录 HTTP 请求日志

### 5.5 验收标准

- [ ] 比例变更前后，同一主播不同时间打赏的分成计算正确
- [ ] 重复接收同一 tip 不产生重复 settlement
- [ ] 余额查询与 settlement 汇总一致

## 六、成员四：经营分析服务 + 模拟服务

### 6.1 经营分析服务（analytics-service）

#### 接口清单

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/analytics/tips/hourly` | 多维打赏分析查询 |
| GET | `/api/analytics/audience/{audienceId}/profile` | 观众消费画像 |

#### 详细要求

1. **多维分析查询**
   - 参数：`anchorId`、`audienceGender`、开始小时、结束小时（如 18~22 点）
   - 返回：每个小时一条汇总（共 5 条），含 `statHour`、`totalAmount`、`tipCount`
   - 数据来自 `tip_hourly_stat`，允许定时任务延迟聚合（如每 5 分钟）

2. **观众画像**
   - 按全站观众累计打赏排名：前 20% 高消费、20%~80% 中消费、后 20% 低消费
   - 更新 `audience_tip_summary.profile_level`
   - 定时任务刷新（可延迟）

3. **定时任务**
   - `TipHourlyStatJob`：从 `tip_record` 聚合写入 `tip_hourly_stat`
   - `AudienceProfileJob`：计算并更新画像等级

### 6.2 模拟服务（mock-service）

#### 职责

模拟观众打赏流量，用于压测与演示。

#### 要求

- 每个节点 **每秒至少 500 次** 打赏请求（可配置）
- 随机选择主播、观众、金额
- 生成唯一 `tipNo`，调用 audience-service 打赏接口
- 支持开关：启动/停止模拟、调整 QPS

#### 接口建议

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/mock/tips/start` | 开始模拟 |
| POST | `/api/mock/tips/stop` | 停止模拟 |
| GET | `/api/mock/tips/status` | 查看当前 QPS 与统计 |

### 6.3 验收标准

- [ ] 18~22 点、指定性别/主播查询返回 5 条小时数据
- [ ] 画像三档划分与 SQL 排名一致
- [ ] mock 单节点稳定达到 500 QPS（本地可调低阈值验证逻辑）
